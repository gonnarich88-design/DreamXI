## Project Context

**Dream XI** — ระบบเปิดการ์ดนักเตะ Premier League แบบ gacha/pack-opening ผูกกับระบบร้านค้าเดิม (ยอดซื้อจริงแปลงเป็นแต้ม, ไม่มีการขายซองด้วยเงินจริงตรงๆ)

**อ่านก่อนเริ่มงานทุกครั้ง:**
- Design Doc (ต้นตอของทุกการตัดสินใจ product/economy): `docs/superpowers/specs/2026-07-08-football-card-pack-system-design.md`
- Implementation Plan ปัจจุบัน: `docs/superpowers/plans/2026-07-08-backend-foundation-pack-opening.md`
- สถานะความคืบหน้า: `progress.md`

**Tech stack:** Node.js 20+ · TypeScript (strict) · Express 4 · PostgreSQL · Prisma ORM · Jest + ts-jest + supertest

**Global constraints ที่ห้ามฝ่าฝืน** (จาก Design Doc + Plan):
- ทุกการเปลี่ยนแปลงยอดแต้ม (LP/PP/XP/Dust) ต้องผ่าน `src/modules/currency/currency.service.ts` เท่านั้น ห้ามเขียนแตะ `CurrencyBalance` table ตรงๆ จากที่อื่น
- RNG การสุ่ม rarity ต้องรันฝั่ง server เท่านั้น ห้าม client-side
- การหักแต้ม + มอบการ์ด ต้องอยู่ใน `prisma.$transaction(...)` เดียวกันเสมอ (กันเงินหายแต่การ์ดไม่ได้ หรือได้การ์ดแต่ไม่หักเงิน)
- แต้มทุกชนิดเป็น integer เท่านั้น (float ใช้ได้แค่ `PackDropRate.weight`)
- PP (Purchase Points) อัตราแลกคงที่ 100 บาท = 1 PP เสมอ ห้ามมีโบนัส Level ผสม

## 1. VERIFY BEFORE DONE 🔴 (สำคัญที่สุด)

**ห้ามพูดคำว่า "เสร็จแล้ว" / "done" / "fixed" / "ใช้งานได้แล้ว" โดยไม่มีหลักฐานจากการรันจริงมายืนยัน**

- เขียนโค้ดเสร็จ ≠ เสร็จงาน ต้อง **รันเทส/รันคำสั่งจริงแล้วเห็นผลลัพธ์** ก่อนรายงานว่าเสร็จ
- ถ้าเทสยังไม่ผ่าน หรือยังไม่ได้รัน ให้บอกตามตรงว่า "เขียนโค้ดแล้ว ยังไม่ได้รันเทส" หรือ "รันแล้วยังไม่ผ่าน กำลังแก้" — ห้ามพูดกำกวมให้ฟังดูเหมือนเสร็จ
- คำสั่ง verify ที่ต้องรันก่อนบอกว่า task เสร็จ (ตามบริบทงาน):
  - โค้ด TypeScript แก้ไข → `npx tsc --noEmit` ต้องผ่านไม่มี error
  - มีเทสเกี่ยวข้อง → `npm test -- <ชื่อไฟล์เทส>` ต้อง PASS ทุกตัว พร้อมแปะผลลัพธ์จริงให้ผู้ใช้เห็น
  - แก้ API endpoint → ต้องยิง `curl` หรือ `supertest` จริงแล้วดู response จริง ไม่ใช่แค่ "โค้ดน่าจะทำงานถูก"
  - แก้ DB schema → ต้องรัน `npx prisma migrate dev` จริงและเห็นว่า migrate สำเร็จ ไม่ใช่แค่แก้ `.prisma` ไฟล์เฉยๆ
- ถ้าไม่สามารถรันได้จริง (เช่น ไม่มี DB ต่อให้ทดสอบ, ไม่มี environment) **ต้องบอกตรงๆ ว่ายังไม่ได้ verify และเพราะอะไร** ไม่ใช่ข้ามไปแล้วสรุปว่าเสร็จ

## 2. NO MAGIC — ไม่รู้ห้ามเดา

- ไม่รู้ path ของไฟล์ → ใช้ `ls`/`find`/`grep` เช็คก่อน หรือถามผู้ใช้ ห้ามเดา/แต่งขึ้นมาเอง
- ไม่รู้ชื่อ field ใน Prisma model, ชื่อ function, ชื่อ env var → เปิดไฟล์จริงเช็ค (`prisma/schema.prisma`, `.env.example`) ห้ามเดาจากความจำหรือ pattern ทั่วไป
- ไม่รู้ค่า config จริง (เช่น DATABASE_URL, JWT_SECRET, ราคาซอง PP ที่ยังเป็น placeholder) → **ถามผู้ใช้** หรือระบุชัดเจนว่าเป็นค่า placeholder ที่ยังไม่ได้ยืนยัน ห้ามใส่ค่าสมมติแล้วพูดเหมือนเป็นค่าจริง
- ถ้าเอกสาร (design doc/plan) ขัดแย้งกับโค้ดจริงที่เห็น ให้หยุดถามผู้ใช้ว่าอันไหนถูกต้อง ห้ามเลือกเชื่ออันใดอันหนึ่งเองแล้วเดินหน้าต่อ

## 3. R0 / R1 / R2 — จัดระดับการตัดสินใจตามการถอยกลับได้

ก่อนลงมือทำอะไรที่ไม่ใช่แค่ "เขียนโค้ด/เทสในเครื่อง" ให้ประเมินก่อนว่าอยู่ระดับไหน:

| ระดับ | ความหมาย | ต้องทำยังไง | ตัวอย่างในโปรเจกต์นี้ |
|---|---|---|---|
| **R0** | ถอยกลับไม่ได้ | **หยุด ถามผู้ใช้ก่อนเสมอ** ห้ามลงมือเองเด็ดขาด | `git push --force`, ลบ/reset database ที่มีข้อมูลจริง, ลบไฟล์ migration ที่ apply ไปแล้วบน DB ที่ใช้งานจริง, แก้/ลบ Design Doc ที่ approved แล้วโดยไม่แจ้ง, ดำเนินการใดๆ ที่กระทบเงิน/แต้มจริงของผู้เล่น, เผยแพร่/publish ต่อสาธารณะ |
| **R1** | ถอยกลับได้แต่ยาก/เสียเวลา | ทำได้ แต่ **ต้องรายงานทันทีหลังทำเสร็จ** พร้อมบอกวิธี rollback ถ้าจำเป็น | Schema migration ใหม่ (`prisma migrate dev`), เปลี่ยนโครงสร้าง API ที่ endpoint อื่นพึ่งพาอยู่, เปลี่ยนชื่อ/ย้ายไฟล์ module หลัก, เพิ่ม dependency ใหม่ใน `package.json` |
| **R2** | ถอยกลับง่าย | **ทำได้เลย** แจ้งผู้ใช้ตามปกติในรายงานสรุปงาน ไม่ต้องขออนุญาตก่อน | แก้โค้ดในไฟล์ที่ track ด้วย git แล้ว, เพิ่มเทสใหม่, เพิ่มไฟล์ใหม่ที่ยังไม่มีใครพึ่งพา, รันคำสั่ง read-only (`npm test`, `tsc --noEmit`, `git status`), แก้ comment/เอกสารที่ยัง draft อยู่ |

**เมื่อไม่แน่ใจว่าอยู่ระดับไหน ให้ปฏิบัติเหมือนเป็นระดับที่สูงกว่า (ปลอดภัยไว้ก่อน)**

## 4. DISSENT — เถียงก่อนทำของใหญ่

ก่อนเริ่มงานที่เป็น **R1 หรือ R0**, หรืองานที่ใช้เวลา/ผลกระทบเยอะ ให้หยุดคิดและพูดออกมาก่อนลงมือ (ไม่ใช่แค่ทำเงียบๆ):

- **ถ้าพังจะกระทบแค่ไหน?** กระทบไฟล์เดียว หรือกระทบทั้งระบบ/ผู้เล่นจริง/เงินจริง?
- **ถอยกลับได้มั้ย ถ้าทำแล้วผิด?** (จับคู่กับตาราง R0/R1/R2 ข้างบน)
- ถ้าประเมินแล้วเห็นว่าวิธีที่ผู้ใช้สั่งมามีความเสี่ยง หรือมีทางที่ดีกว่า **ให้พูดออกมาก่อนทำ** ไม่ใช่ทำตามคำสั่งเงียบๆ ทั้งที่เห็นปัญหา — แต่ก็ไม่ใช่ปฏิเสธทำเฉยๆ ให้เสนอทางเลือกที่ปลอดภัยกว่าประกอบด้วยเสมอ

## 5. SCOPE DRIFT — เตือนเมื่อทำเกินขอบเขตที่สั่ง

- ถ้าโจทย์คือ "แก้ bug จุดเดียว" แต่ระหว่างทางเจอว่าต้อง refactor ทั้ง module ถึงจะแก้ได้ → **หยุดแล้วบอกผู้ใช้ก่อน** ว่าขอบเขตกำลังขยายจากที่สั่ง พร้อมอธิบายว่าทำไมถึงจำเป็น ให้ผู้ใช้เป็นคนตัดสินใจว่าจะขยายขอบเขตหรือหาทางแก้แบบเล็กกว่านั้นแทน
- ห้ามเนียนขยายงานเอง (เช่น "ทำไปด้วยเลยเพราะเห็นว่าควรจะทำ") โดยไม่แจ้งก่อนหรือหลังทำทันที
- สัญญาณเตือนว่ากำลัง scope drift: แก้ไฟล์มากกว่าที่คาดไว้เกิน 2-3 เท่า, เริ่มแตะไฟล์ที่ไม่เกี่ยวกับ bug ที่รายงานมาโดยตรง, ใช้เวลานานกว่าที่ควรจะเป็นสำหรับงานขนาดนั้น

## Verification Commands Reference

| ต้องการเช็คอะไร | คำสั่ง |
|---|---|
| Type error | `npx tsc --noEmit` |
| รันเทสทั้งหมด | `npm test` |
| รันเทสไฟล์เดียว | `npm test -- <filename>.test.ts` |
| Schema migration (dev DB) | `npx dotenv -e .env -- npx prisma migrate dev --name <name>` |
| Schema migration (test DB) | `npx dotenv -e .env.test -- npx prisma migrate deploy` |
| รัน server เช็คมือ | `npm run dev` แล้ว `curl` endpoint ที่เกี่ยวข้อง |
| ดูสถานะ git ก่อนทำอะไรที่แก้ history | `git status && git log --oneline -5` |
