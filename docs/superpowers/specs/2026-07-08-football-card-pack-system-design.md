# Design Doc: Dream XI — ระบบเปิดการ์ดนักเตะ (Football Player Card Pack-Opening System)

**ชื่อโปรเจกต์/เกม:** Dream XI
**วันที่:** 2026-07-08
**สถานะ:** Approved — ผ่านการ brainstorm และแก้ไขจุดอ่อนร่วมกับผู้ใช้แล้ว พร้อมเข้าสู่ implementation planning

> ⚠️ ก่อนใช้ชื่อ "Dream XI" เป็นทางการ แนะนำให้ตรวจสอบว่าไม่ชนกับเครื่องหมายการค้า/ชื่อเกมที่มีอยู่แล้วในตลาด (เช่น App Store, Play Store, กรมทรัพย์สินทางปัญญา) ก่อนจดทะเบียนแบรนด์จริง

---

## 1. ภาพรวมโปรเจกต์

ระบบเปิดการ์ดนักเตะแบบ gacha/pack-opening (แนวเดียวกับ FIFA Ultimate Team) โดยเป็น **ฟีเจอร์ gamification ที่ผูกกับธุรกิจร้านค้า/แพลตฟอร์มที่มีอยู่แล้ว** ไม่ใช่เกมแยกที่ขายซองด้วยเงินจริงโดยตรง — ผู้เล่นได้แต้มเปิดซองจาก (ก) การมีส่วนร่วมในเกม (ล็อกอิน/ภารกิจ) และ (ข) ยอดซื้อสินค้าจริงจากร้านค้าที่ผูกอยู่

## 2. ⚠️ ข้อควรระวังทางกฎหมาย (สำคัญที่สุด — ต้องดำเนินการคู่ขนานกับการพัฒนา)

### 2.1 ความเสี่ยงเรื่องกฎหมายการพนัน/รางวัลเสี่ยงโชค (ระดับวิกฤต 🔴)

ระบบนี้มีโครงสร้าง **"ซื้อสินค้าจริง → แปลงเป็นแต้ม → สุ่มเปิดได้ของรางวัล"** ซึ่งเข้าข่ายกิจกรรมที่ควบคุมโดยประกาศกระทรวงมหาดไทยเรื่องการเล่นแชร์แถมพกหรือรางวัลด้วยวิธีเสี่ยงโชค (ภายใต้ พ.ร.บ. การพนัน พ.ศ. 2478) ธุรกิจที่จัดโปรโมชั่นลักษณะ "ซื้อครบ X บาท รับสิทธิ์สุ่มของรางวัล" ในไทย **ต้องขอใบอนุญาตจากกรมการปกครองก่อนจัดกิจกรรม**

**Action required: ต้องปรึกษาทนายความ/หน่วยงานที่เกี่ยวข้องก่อนเปิดใช้งานระบบจริง ไม่ใช่แค่การตั้งค่า Terms of Service** — ความเสี่ยงนี้สำคัญกว่าประเด็นลิขสิทธิ์นักเตะด้านล่าง

### 2.2 ความเสี่ยงเรื่องลิขสิทธิ์/สิทธิ์ในตัวตนนักเตะ (Right of Publicity)

- ตัดสินใจใช้ **ชื่อนักเตะจริง + สถิติสาธารณะ** แต่ **ไม่ใช้รูปภาพจริง** (ใช้ภาพประกอบที่ทีมสร้างเอง) เพื่อลดความเสี่ยงด้านลิขสิทธิ์ภาพ
- อย่างไรก็ตาม การใช้ **ชื่อจริง** ของนักเตะดังระดับโลก (Premier League) เพื่อการค้าที่ผูกกับการกระตุ้นยอดขาย ยังมีความเสี่ยงด้าน right of publicity/endorsement ในบางเขตอำนาจศาล
- **Action required:** ปรึกษาทนายด้านทรัพย์สินทางปัญญาก่อน launch จริง

### 2.3 ความเสี่ยงด้านการคุ้มครองเยาวชน

กลไกสุ่มรางวัล (loot box/gacha) กำลังถูกจับตาทั่วโลก — PEGI จะบังคับใช้เรทขั้นต่ำ 16+ สำหรับเกมที่มีกลไกสุ่มไอเทมแบบเสียเงิน/ของมีค่า ตั้งแต่มิถุนายน 2026 หากกลุ่มเป้าหมายมีเยาวชนปนอยู่ ควรพิจารณาระบบยืนยันอายุ

## 3. สถาปัตยกรรมแพลตฟอร์ม

**เป้าหมาย:** Web App เดียวที่ทำงานได้ทั้ง Browser ปกติ, PWA, และ Telegram Mini App โดยไม่ต้องแยกโค้ดเบส

- **Frontend:** SPA แบบ mobile-first responsive
- **Backend:** REST/GraphQL API แยกอิสระจากแพลตฟอร์ม (platform-agnostic) ไม่ผูกกับ Telegram ที่ระดับ business logic
- **Auth layer แบบ pluggable:** เว็บใช้ email/OAuth ปกติ, Telegram Mini App ใช้ Telegram Login Widget + validate `initData` (ตรวจ hash signature ฝั่ง server)
- **Payment/Integration layer แบบ pluggable:** เชื่อมต่อกับระบบร้านค้าเดิมผ่าน webhook (ดูหัวข้อ 6)

Telegram Mini App คือ web app ที่รันใน WebView พร้อมฉีด Telegram WebApp JS SDK เป็นเลเยอร์เสริม — ถ้าออกแบบสถาปัตยกรรมถูกตั้งแต่แรก จะได้โค้ดเบสเดียวใช้ได้ทั้งหมด

## 4. ขอบเขตข้อมูลนักเตะ (Player Data Scope)

- **ลีก:** Premier League เท่านั้น (ประมาณ 20 ทีม x ~25 นักเตะ = ~500 นักเตะ รวม squad สำรอง)
- **ข้อมูลที่ใช้:** ชื่อจริง + สถิติสาธารณะ (ผลงาน, ตำแหน่ง, ค่าสถิติทั่วไป)
- **ภาพ:** ภาพประกอบที่ทีมสร้างเอง ไม่ใช้ภาพถ่ายจริง
- **Open Item:** ยังไม่ได้เลือกแหล่งข้อมูลสถิติ (เช่น API-Football, Sportmonks) และความถี่ในการอัปเดตข้อมูล (transfer window, จบซีซัน) — ต้องตัดสินใจก่อนเริ่ม dev จริง

## 5. Core Gameplay Loop (แบ่งเป็นเฟส)

Collection เป็นแกนกลางที่ทุกโหมดใช้ร่วมกัน แบ่งการพัฒนาเป็นเฟสเพื่อไม่ให้ scope ใหญ่เกินไปในรอบแรก:

- **เฟส 1 (MVP):** เปิดซอง → เก็บการ์ด → จัดทีม 11 คน → แข่งกับบอท/ผู้เล่นอื่นแบบจำลองผล (auto-calculate จากค่าสถิติการ์ด ไม่ใช่เกมเรียลไทม์ คล้าย FUT Weekend League)
- **เฟส 2:** Fantasy layer — จัดทีมรับแต้มตามผลงานจริงของนักเตะในสนาม (ต้องมี data feed สถิติแมตช์จริง — ผูกกับ Open Item ในหัวข้อ 4)
- **~~เฟส 3: Trading/Marketplace~~ — ตัดออกแล้ว** แทนที่ด้วย Fusion System (ดูหัวข้อ 9) เพื่อลดความเสี่ยงเรื่อง scam/บอทเทรด/real-money trading

**Open Item:** สูตรคำนวณผลการแข่งขัน (Match Simulation Engine) ยังไม่ได้ออกแบบรายละเอียด — เป็นงานออกแบบแยกต่างหากในเฟส 1

## 6. ระบบแต้ม (Currency System)

แบ่งเป็น 2 สกุลแยกกันเด็ดขาด เพื่อคุม incentive ให้ตรงเป้าหมายธุรกิจ (กระตุ้นทั้งการมีส่วนร่วมและยอดซื้อ):

| สกุลแต้ม | ที่มา | อัตราแลก | ใช้เปิดซองระดับไหน |
|---|---|---|---|
| **Login Points (LP)** | ล็อกอินรายวัน + streak bonus + ภารกิจรายวัน/รายสัปดาห์ + ชนะแมตช์จำลอง | ไม่คงที่ (ปรับ balance ได้) | Bronze, Silver |
| **Purchase Points (PP)** | ยอดซื้อสินค้าจริงจากร้านค้าที่ผูกอยู่ | **คงที่ตายตัว 100 บาท = 1 PP ทุกกรณี ไม่มีข้อยกเว้น** | Gold, Special/Icon |

**หลักการสำคัญ:**
- **PP ไม่รับโบนัส % จาก Level ใดๆ ทั้งสิ้น** — อัตราแลกต้องคงที่เพื่อความโปร่งใสทางกฎหมาย/บัญชี และง่ายต่อการคำนวณ clawback (หัวข้อ 12)
- **LP รับโบนัส % ได้ตาม Level** เพราะเป็นแต้มฟรี ไม่กระทบประเด็นกฎหมาย/บัญชี

### 6.1 การเข้าถึงซอง (Pack Access)

**หลักการ: ราคาซอง (จำนวนแต้มที่ต้องใช้) เป็นตัวกั้นเพียงอย่างเดียว ไม่มี Level-lock ซ้อนทับ** — เพื่อไม่ให้ลูกค้าที่จ่ายเงินจริงรู้สึกว่าจ่ายแล้วใช้แต้มไม่ได้ทันทีเพราะ Level ไม่ถึง

**ราคาซอง (Config-driven — TBD ตัวเลขจริง):**

| ซอง | ราคา (ตัวอย่าง placeholder) | ชนิดแต้มที่ใช้ได้ |
|---|---|---|
| Bronze | 10 LP | LP |
| Silver | 45 LP | LP |
| Gold | TBD PP | PP เท่านั้น |
| Special/Icon | TBD PP | PP เท่านั้น |

> **Open Item:** ราคาซองฝั่ง PP (Gold/Special) ต้องคำนวณจากยอดซื้อเฉลี่ยต่อออเดอร์จริงของร้านค้า — ยังไม่มีข้อมูลนี้ ณ วันที่เขียน spec นี้ ให้ใช้ placeholder ระหว่างพัฒนา แล้วปรับจริงก่อน launch โดยเก็บเป็นค่าที่ปรับได้ (config) ไม่ hardcode ในโค้ด

## 7. ระบบ Drop Rate (เปอร์เซ็นต์การสุ่ม)

- แต่ละใบในซองสุ่ม**อิสระจากกัน** (independent event ต่อการเปิดแต่ละครั้ง)
- Rarity tier: Bronze → Silver → Gold → Special/Icon (ตั้งชื่อ/จำนวน tier ให้ตรงกับที่ทีมต้องการปรับได้)
- แสดง % แบบ**ต่อกลุ่ม tier** ไม่ใช่ต่อนักเตะรายคน (เพราะแต่ละ tier มีนักเตะหลายสิบ-หลายร้อยคน) — แม้ระบบนี้เป็น "ฟรีทั้งหมด" ไม่มีข้อบังคับทางกฎหมายให้เปิดเผย % แต่**แนะนำให้แสดง % อยู่ดี** เพื่อความโปร่งใสกับผู้เล่นและมาตรฐานอุตสาหกรรม
- **ไม่มี Soft Pity ใน V1** — ใช้ Hard Pity เท่านั้น (ดูหัวข้อ 8) เพื่อลดความซับซ้อนของ RNG formula ช่วงแรก

## 8. ระบบ Pity (การันตีของหายาก)

**นับแยกอิสระต่อประเภทซอง (per pack-type) สะสมตลอดอายุบัญชี**

| ประเภทซอง | Pity threshold (ตัวอย่าง) | การันตีอะไร |
|---|---|---|
| Bronze | ไม่มี pity | — |
| Silver | ทุก 20 ครั้ง | การ์ด Rare-tier อย่างน้อย 1 ใบ |
| Gold | ทุก 30 ครั้ง | การ์ด Epic-tier อย่างน้อย 1 ใบ |
| Special/Icon | ทุก 10 ครั้ง | การ์ด Icon-tier อย่างน้อย 1 ใบ |

**กติกา:**
1. Pity การันตีแค่ "ระดับความหายาก" ไม่การันตีนักเตะเจาะจง — ยังสุ่มภายในพูลของ tier ตามปกติ
2. เคาน์เตอร์รีเซ็ตเป็น 0 ทันทีที่ pity ทำงาน ไม่สะสมต่อ
3. **ไม่รีเซ็ตด้วยเหตุผลอื่น** — ไม่ผูกกับ Level, ไม่รีเซ็ตตามเวลา, ไม่รีเซ็ตแม้ผู้เล่นหายไปนาน (เก็บถาวรในฐานข้อมูลผูกกับ account + pack-type)

## 9. การจัดการการ์ดซ้ำ (Duplicate Handling)

ไม่มี P2P Marketplace (ตัดออกเพื่อลดความเสี่ยง scam/บอทเทรด/real-money trading) — แทนที่ด้วย 2 ระบบที่ผู้เล่นเลือกเองได้:

### 9.1 Disenchant → Dust

| การ์ดซ้ำระดับ | ได้ Dust |
|---|---|
| Bronze | 5 |
| Silver | 20 |
| Gold | 100 |
| Special/Icon | 500 |

### 9.2 Fusion System

รวมการ์ดซ้ำ **10 ใบระดับเดียวกัน** → การันตีได้การ์ด **1 ใบระดับถัดไป** (สุ่มภายในพูลของ tier ถัดไป)
- เทียร์บนสุด (Special/Icon) ไม่มีระดับถัดไป → ใช้ Fusion แบบ reroll ภายในเทียร์เดิม (รวม Special ซ้ำ → สุ่ม Special ใบใหม่ที่ยังไม่มี)
- **ใช้งานได้ตั้งแต่ Level 1** (ไม่มีความเสี่ยงเรื่องโกงจากผู้เล่นคนอื่น เพราะเป็นระบบภายในบัญชีตัวเอง)
- Level 21+ (Elite tier) ได้ **Fusion Discount** — ใช้แค่ 8 ใบแทน 10 ใบ

### 9.3 Dust Shop (ควบคุมเข้มงวดเพื่อกันเฟ้อ)

รายได้ Dust เฉลี่ย ~25-30/วัน สำหรับผู้เล่น active สม่ำเสมอ

| สินค้า | ราคา (Dust) | ระยะเวลาสะสม (ประมาณ) |
|---|---|---|
| การ์ด Silver เลือกเองได้ (หมุนเวียนรายสัปดาห์) | 300 | ~10-12 วัน |
| การ์ด Gold สุ่ม (ไม่เลือกตัว) | 2,000 | ~2-3 เดือน |
| **Special/Icon — ห้ามขายเด็ดขาด** | — | รักษาความหายากของเทียร์สูงสุด |

**กลไกป้องกันเฟ้อเพิ่มเติม:**
- รายการ Silver ในชอปหมุนเวียนทุกสัปดาห์ (ไม่ใช่ list ตายตัว)
- Gold token เป็นแบบสุ่มเท่านั้น ไม่เลือกตัว
- **จำกัดซื้อ Gold token ได้ 1 ครั้ง/เดือน/บัญชี** แม้จะมี Dust พอซื้อมากกว่านั้น

## 10. ระบบ User Level

**30 Level แบ่งเป็น 5 Tier (Tier ละ 6 Level)** — Level คุมเฉพาะ "ฟีเจอร์ที่ปลดล็อก" และ "โบนัส LP" เท่านั้น ไม่แตะการเข้าถึงซอง (แยกจากหัวข้อ 6.1 อย่างเด็ดขาด)

| Tier | Level | ฟีเจอร์ที่ปลดล็อก | โบนัส (เฉพาะ LP) |
|---|---|---|---|
| 🟫 Rookie | 1–5 | สร้างทีมพื้นฐาน, Fusion (10 ใบ) | — |
| ⚪ Amateur | 6–10 | — | LP +10% |
| 🟡 Semi-Pro | 11–15 | Match Simulation | LP +20% |
| 🔵 Professional | 16–20 | Leaderboard/Ranking | LP +20% |
| 🟣 Elite | 21–25 | Fusion Discount (8 ใบ) | LP +30% |
| 🟠 Legend | 26–30 | Badge เกียรติยศ | LP +30% |

**แหล่งที่มาของ XP (แยกจากแต้ม LP/PP — คนละระบบ):**

| กิจกรรม | ได้ XP |
|---|---|
| ล็อกอินรายวัน (ตาม streak) | ✅ |
| ทำภารกิจรายวัน/รายสัปดาห์สำเร็จ | ✅ |
| ชนะแมตช์จำลอง (เฟส 1) | ✅ |
| **เปิดซอง** | ❌ ไม่ได้ XP (ได้แค่การ์ด) |

**เหตุผลที่ตัด XP จากการเปิดซอง:** ป้องกันไม่ให้คนจ่ายเงินซื้อ PP แล้วเปิดซองรัวๆ ได้เปรียบเรื่องความเร็วในการไต่ Level เหนือผู้เล่นกลุ่ม engagement สูงที่ไม่จ่ายเงิน — Level ควรวัด "ความขยัน/การมีส่วนร่วม" ล้วนๆ แยกจาก "ทรัพยากรที่ใช้เปิดซอง"

**Max Level (30):** แนะนำใส่ Prestige system เป็น future enhancement (ไม่ทำใน MVP) — รีเซ็ตกลับ Level 1 แลก badge สะสมพิเศษ รองรับผู้เล่นระยะยาว

## 11. Purchase Points Lifecycle (Pending → Confirmed) และ Clawback

เพื่อจัดการกรณีลูกค้าคืนสินค้า/ยกเลิกออเดอร์หลังได้ PP ไปแล้ว:

1. ลูกค้าซื้อของ → รับ webhook `purchase.completed` จากร้านค้า → สร้าง PP สถานะ **Pending** (โชว์ในโปรไฟล์ แต่ใช้เปิดซองไม่ได้)
2. รอผ่าน **Return Window ของร้านค้า** (เช่น 7-14 วัน ตาม policy คืนสินค้าปกติ) โดยไม่มีการคืนสินค้า → เปลี่ยนเป็น **Confirmed** อัตโนมัติ (ใช้เปิดซองได้)
3. ถ้ามีการคืนสินค้าระหว่าง Pending (รับ webhook `purchase.refunded`) → ลบ PP ที่ pending อยู่ทิ้ง ไม่กระทบผู้เล่นเพราะยังไม่เคยใช้ได้

**Edge case — คืนสินค้าหลัง PP confirmed แล้ว (อาจถูกใช้เปิดซองไปแล้ว):**
- หัก PP คงเหลือก่อน ถ้าไม่พอให้ติดลบ (PP balance ติดลบชั่วคราวได้)
- ยอดติดลบ → ระงับสิทธิ์รับ/ใช้ PP ใหม่จนกว่ายอดจะกลับเป็นบวก
- **ไม่เรียกคืนการ์ดที่ได้ไปแล้ว** — สร้างประสบการณ์แย่และตรวจสอบยาก

**ต้องมีทางเทคนิค:** webhook 2 ทาง (`purchase.completed`, `purchase.refunded`) จากระบบร้านค้าเดิม + cron job เปลี่ยนสถานะ Pending → Confirmed อัตโนมัติ + ledger บันทึกทุก transaction พร้อม order reference ID

## 12. Anti-Fraud พื้นฐาน (Baseline)

**Structural (ต้องฝังไว้ตั้งแต่แรก):**
1. **1 บัญชีร้านค้า = ผูกได้กับ 1 บัญชีเกมเท่านั้น**
2. **ยืนยันตัวตนขั้นต่ำตอนสมัคร** (เบอร์โทร/อีเมล ผ่าน OTP)

**Open Item (ทำหลัง launch ตามข้อมูลการโกงจริงที่พบ):**
- Device fingerprinting / IP rate-limiting แบบละเอียด
- ระบบตรวจจับพฤติกรรมโกงอัตโนมัติ (ML-based abuse detection)
- นโยบายจำกัดจำนวนบัญชีต่อ IP/device

## 13. สถาปัตยกรรมเทคนิคที่ต้องมี

- **RNG Engine ฝั่ง server เท่านั้น** (ห้าม client-side สุ่ม) พร้อม seed/audit log ย้อนหลังได้
- **Ledger/Transaction system** สำหรับ LP, PP, Dust, XP — ป้องกัน duplicate/race condition เวลาเปิดซองพร้อมกันหลาย request
- **Inventory system** เก็บการ์ดที่ผู้เล่นมี + Card Catalog (ฐานข้อมูลนักเตะ, ค่าสถิติ, ภาพประกอบ)
- **Pity counter storage** แยกต่อ pack-type ต่อ account (ถาวร)
- **Webhook integration** กับระบบร้านค้าเดิม (`purchase.completed`, `purchase.refunded`)
- **Cron job** สำหรับเปลี่ยนสถานะ PP Pending → Confirmed
- **Anti-cheat/rate-limiting** พื้นฐานระดับ API
- **Odds/probability audit log** เก็บสถิติเปิดจริงเทียบกับ % ที่ประกาศไว้ (best practice แม้ไม่มีข้อบังคับทางกฎหมายเพราะไม่มีเงินจริงในระบบซองโดยตรง)

## 15. สรุป Decision Log (สิ่งที่ตัดสินใจแล้วระหว่าง brainstorm)

- โมเดลแต้ม: ฟรีทั้งหมด ไม่มีการขายซองด้วยเงินจริงตรงๆ (ทางเลือกที่ 2 จาก 3 ทางเลือก)
- Platform: Web app เดียว รองรับ Browser/PWA/Telegram Mini App
- Player data: ชื่อจริง + สถิติสาธารณะ, ภาพประกอบเอง (ไม่ใช้รูปจริง)
- League scope: Premier League เท่านั้น
- Core loop: Collection + Match Simulation (เฟส 1) → Fantasy (เฟส 2) — ตัด P2P Marketplace ออก แทนด้วย Fusion System
- Currency: แยก LP/PP เด็ดขาด, PP อัตราคงที่ 100 บาท = 1 PP ตลอด (ไม่มีโบนัส Level)
- Pack access: กั้นด้วยราคาแต้มเท่านั้น ไม่ผูก Level
- Level: 30 levels/5 tiers, XP ไม่รวมการเปิดซอง, โบนัส Level ใช้กับ LP เท่านั้น
- Pity: Hard pity เท่านั้นใน V1, แยกนับต่อ pack-type ตลอดชีพ
- Duplicate: Disenchant→Dust + Fusion (10 ใบ→1 ใบเทียร์ถัดไป), ไม่มี P2P trading
- Dust Shop: จำกัดขาย Silver(เลือกเอง)/Gold(สุ่ม)เท่านั้น ห้ามขาย Special/Icon, จำกัดซื้อ Gold 1 ครั้ง/เดือน
- PP lifecycle: Pending→Confirmed ผ่าน return window, มี clawback สำหรับ edge case
- Anti-fraud baseline: ผูก 1 บัญชีร้านค้า:1 บัญชีเกม + OTP verification
